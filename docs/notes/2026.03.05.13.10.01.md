# MCP Lith Nexus Server (v0.1)

## Priority
- high

## Complexity
- high (protocol surface + indexing + deterministic identity + idempotent fact store)

## Intent
Build an MCP server that exposes the repo’s knowledge graph (canonical `nexus_graph`) to LLMs, using **Lith (s-expr)** as the primary content payload (instead of JSON).

The server must let an LLM:
1) **Find** nexus resources (search)
2) **Read** nexus resources (full record)
3) **Create** nexus resources *as PromptDB-compatible Lith facts* (append-only fact store + idempotence)

This is a *local-agent* continuity mechanism: the repo + git history are source of truth.

## Non-goals
- Do not replace MCP’s JSON-RPC envelope; only replace *payload semantics* (return/accept Lith strings).
- Do not require the full UI/simulation to be running.
- Do not introduce a new graph schema: reuse canonical `nexus_graph` semantics.

## Definitions

### Lith payload
A textual s-expression payload, supporting:
- `;` comments
- lists: `( ... )`
- vectors: `[ ... ]`
- maps: `{ :k v :k2 v2 }`
- strings: "..."

### Nexus resource
An entity that is addressable and stable across runs:
- `id` (deterministic)
- `kind` (role/type; e.g. `file`, `spec`, `url`, `concept`, `field`, `anchor`)
- attributes (label/path/url/tags/summary/etc)
- edges to other entities

### PromptDB Lith facts
Append-only Lith forms compatible with your fact/obs/q semantics, e.g.
- `(fact (ctx 世) (claim (...)) (source "...") (p 0.8) (time "2026-03-05T...Z"))`
- `(obs  (ctx 汝) (about ...) (signal ...) (p 0.7) (time ...) (source ...))`
- `(q    (ctx 己) (ask "...") (why-blocked "..."))`

(Exact “claim” shapes are schema’d below.)

## Core requirements

### R1 — MCP serves Lith, not JSON
All MCP tool outputs and resource reads must return content as Lith text.

Transport remains MCP standard, but the semantic payload is:
- `mimeType: text/lith` (preferred) or `text/plain` with `lith` convention
- `text: "<lith s-expr here>"`

### R2 — Deterministic IDs
IDs must be stable across sessions and machines.

Canonical ID rules:
- File entity: `id = sha1("file:" + repoRelPosixPath)`
- Spec entity: `id = sha1("spec:" + repoRelPosixPath)`
- URL entity: `id = sha1("url:" + normalizedUrl)`
- Concept entity: `id = sha1("concept:" + slug(label))` unless explicitly provided

Normalization rules:
- repoRelPosixPath: repo-relative, forward slashes, no `./`, no `..`
- normalizedUrl: lowercase scheme/host, strip default ports, preserve path/query

### R3 — Graph sources (offline-first)
The server must build an offline “nexus view” from repo artifacts:

Minimum sources:
- `specs/**/*.md` (spec nodes)
- `.opencode/promptdb/**/*` (prompt fragments/contracts as nodes)
- `.opencode/runtime/eta_mu_knowledge.v1.jsonl` (archived knowledge entries as nodes)
- `receipts.log` (events/refs as edges/annotations where useful)

Optional live source (Phase 5):
- If a simulation/runtime endpoint is available, it may be used as a *secondary* source for freshness, but must not be required.

### R4 — Find / Read / Create

#### Find
LLM can search by:
- text query (label/path/url/tags)
- kind filter
- time window (from receipts or index metadata)
Return: ranked candidates in Lith.

#### Read
LLM can request a full entity by `id` or URI.
Return: canonical Lith record including:
- entity attributes
- outbound edges (typed)
- attached facts/observations/questions relevant to that entity

#### Create (as PromptDB Lith facts)
LLM can create/attach meaning by writing facts, not by mutating canonical sources directly.

Create supports:
- Add/attach attributes via facts (e.g. tag, summary, field-score)
- Add edges via facts (e.g. “spec references file”, “file supports concept”)
- Create a new concept/anchor entity (optional, but must be deterministic)

## In-repo storage: fact log + registry (append-only)

### Fact store
Write new facts to one append-only location:

Preferred:
- `.opencode/runtime/nexus_facts.v1.lith` (append-only)
- `.opencode/runtime/nexus_facts.registry.jsonl` (idempotence registry)

Alternate (if you want PromptDB locality):
- `.opencode/promptdb/facts/v1/*.lith` (date-bucketed append files)
- plus registry in `.opencode/runtime/`

### Idempotence
Registry key:
- `sha256(canonical_printed_fact_form)`

If a fact already exists (same digest), treat as no-op and return `(ok {:status :no-op ...})`.

### Safety rails
- Only allow writes within the approved fact-store paths.
- Refuse to write if the payload appears to contain secrets (basic detectors; if triggered, return `(err ...)` and require explicit override outside this tool).

## Schema: Lith records returned by server

### `nexus/node.v1`
```lisp
(node.v1
  { :id "sha1:..."
    :kind :file|:spec|:url|:concept|:field|:anchor|:prompt_fragment|:receipt
    :label "..."
    :path "specs/drafts/..."
    :url "https://..."
    :tags ["..."]
    :summary "..."
    :attrs { :k v ... }
    :sources ["specs/..." ".opencode/runtime/..." ...]
    :edges [ (edge.v1 { ... }) ... ]
    :facts [ (fact ...) (obs ...) (q ...) ... ] })
````

### `nexus/edge.v1`

```lisp
(edge.v1
  { :id "sha1:edge:..."
    :src "sha1:..."
    :rel :references|:links_to|:categorized_as|:supports|:derived_from|:mentions
    :dst "sha1:..."
    :p 0.0
    :time "2026-03-05T00:00:00Z"
    :source "..." })
```

### `nexus/search_results.v1`

```lisp
(results.v1
  { :query "..."
    :kinds [:spec :file]
    :hits
      [ { :id "sha1:..." :kind :spec :label "..." :path "..." :score 0.93 }
        ... ] })
```

### `nexus/error.v1`

```lisp
(err.v1
  { :code :parse_error|:not_found|:forbidden|:io|:secret_suspected
    :message "..."
    :where { :file "..." :line 12 :col 8 } })
```

## MCP surface (tools + resources)

### Tools (must exist)

1. `nexus.search`

* args: `{ query: string, kinds?: string[], limit?: number }`
* returns: `(results.v1 ...)`

2. `nexus.read`

* args: `{ id?: string, uri?: string }`
* returns: `(node.v1 ...)` or `(err.v1 ...)`

3. `nexus.fact.append`

* args: `{ lith: string }` (one or many fact forms)
* behavior: validate → canonical print → sha256 → append + registry
* returns: `(ok.v1 { :status :appended|:no-op :digests [...] })` or `(err.v1 ...)`

4. `lith.parse`

* args: `{ lith: string }`
* returns: `(ok.v1 { :forms N :summary [...] })` or `(err.v1 ...)`

### Resources (read-only, but required)

* `lith://schema/nexus/node.v1`
* `lith://schema/nexus/edge.v1`
* `lith://schema/promptdb/fact.v1`
* `lith://index/specs` (summary list of spec nodes)
* `nexus://node/<id>` (alias of `nexus.read`)

## Parsing specs into nexus nodes (offline indexer)

Index `specs/**/*.md`:

* derive `node.kind = :spec`
* parse YAML frontmatter if present
* derive edges:

  * `:references` edges for any paths in “Candidate Files”
  * `:mentions` edges for detected `part64/code/...` strings
* derive facts:

  * `(fact (ctx 世) (claim (spec/priority <id> :high)) ...)`
  * `(fact (ctx 世) (claim (spec/status <id> :inprogress)) ...)`
  * `(q ...)` extracted from “Open Questions” sections (optional)

All derived facts must be written through `nexus.fact.append` so the fact store remains the single write path.
